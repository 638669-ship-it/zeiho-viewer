# 税務六法 Web（zeiho-viewer）

関根稔弁護士作成「国税法令＆通達集」（税務六法アウトライン表示ソフト）と同等の機能を、
現代のWeb技術で再構築するプロジェクト。税理士業務中に iPad・PC のブラウザから
条文・通達を即座に参照するためのツール。

- 発注者：福島会計事務所 税理士 福島健人
- 画面デザイン正本：`税務六法Web_表示イメージ.html`（v3カラフル版）
- 着手指示書：`税法条文ビューア_着手指示書_v2.md`

## 現在の状況

**Phase 1（法令ビューア）完了。** 24法令・本則7,169条を収録し、ビューアが動作する。

| Phase | 内容 | 状態 |

| 1 | パーサ＋ビューア公開（本丸） | 完了（公開設定のみ未実施） |

## 使い方

Python 3.10+ のみ。追加パッケージ不要（標準ライブラリだけで動く）。

```powershell
python scripts/fetch_laws.py        # e-Gov から24本の全文JSONを data/raw/ に取得（約97MB・約1分）
python scripts/verify_raw.py        # 取得結果の検収（条数・ランドマーク条文）
python scripts/parse_laws.py        # 階層JSONへ変換し docs/data/ に出力（約64MB・約1分）
python scripts/build_local_zip.py   # オフライン版 zip を dist/ に出力（約8MB）

# ビューアをローカルで見る（file:// では fetch がブロックされるので必ずHTTPで開く）
python -m http.server 8000 --directory docs
# → http://127.0.0.1:8000/
```

`data/raw/` は再取得可能なため git 管理外。`docs/data/` は GitHub Pages が配信する
実体なのでコミットする。

## データ設計（Phase 2 への申し送り）

### 分割とロード戦略

**附則が全体の約半分を占める。** 所得税法は16.4MBのうち本則はわずか0.79MB
（残りは附則12.85MB＋別表1.97MB）。そこで本則・附則・別表を別ファイルに分け、
通常の閲覧では本則しか読まない構成にした。

読み込みは3段階：

| ファイル | 内容 | いつ読むか |
|---|---|---|
| `data/index.json` | 法令一覧・色・略称辞書（8KB） | 起動時 |
| `data/{key}/toc.json` | 目次ツリー＋条→チャンク索引 | 法令を開いた時 |
| `data/{key}/main-NN.json` | 本則の条文本体（350KB目安で分割） | 開いた条を含む1本だけ |
| `data/{key}/suppl.json` / `suppl-NN.json` | 附則（索引＋本体） | 附則タブを開いた時 |
| `data/{key}/appdx.json` | 別表・様式 | 別表タブを開いた時 |

条がどのチャンクにあるかは `toc.json` が持つので、チャンク境界は章に揃えなくてよい。
GitHub Pages は gzip 配信するため、350KBのチャンクは実測 **45KB** で届く。

### e-Gov データの実仕様（実データで確認済み）

1. **`Article/@Num` は3形式**：`"22"`／`"22_2"`（枝番）／`"4:7"`（第四条から第七条まで。約100件）。
   表示は `第22条の2`・`第4条〜第7条`（`第22の2条` ではない）。
2. **条番号は法令内で一意ではない。** 附則にも第1条がある（法人税法：本則262条に対し全文1,111条）。
   条番号ジャンプは `MainProvision` に限定し、附則は改正法令（`@AmendLawNum`）ごとに分けて持つ。
3. **条見出し（`ArticleCaption`）は全条にはない。** → 下記の通り補完した。
4. `Sentence/@Function` は `main`／`proviso`（ただし書）の2値。印刷六法同様に地続きに組む。
5. 定義規定（法法2等）は `ItemSentence > Column` の二段組み。
6. 様式（`AppdxStyle`）の中身は画像（`Fig`）で、`attached_files_info` は null。
   表題と関係条文だけ載せ、原文リンクへ送っている。

### Phase 1 で決めた仕様（指示書の宿題への回答）

**条見出しが無い条の表示**（指示書に「款・節名で代替するか空にするか決めること」とあった件）

→ **直近の節・款・目の名称を見出しとして借り、借り物であることを表示上区別する。**

実データを見ると、見出しの欠落は「その条だけ見出しが無い」のではなく
「款・目の名称がそのまま見出しの役割を負っている」ケースがほとんどだった：

| 条 | 原文の条見出し | 直近の款・目 |
|---|---|---|
| 法法22 | なし | 第二款「各事業年度の所得の金額の計算の通則」 |
| 法法22の2 | なし | 第一目「収益の額」 |
| 措法66の6 | なし | 第一款「内国法人の外国関係会社に係る所得の課税の特例」 |

いずれも印刷六法で当該条の見出しとして掲げられているものと一致する
（モックHTMLが法法22に手書きしていた見出しとも一致した）。編・章は範囲が広すぎて
誤読を招くため使わない。この補完により、見出しが無い条は 500条 → **147条（2.1%）** に減った。
借りた見出しは目次では斜体、条文画面では点線の下線＋ツールチップで区別する
（データ上は `capd:1`）。

## 括弧書きの色分け（指示書9）について

((depth-1)%5)+1 の5色循環という仕組みは仕様どおり。**色そのものは 2026-07-15 に
発注者指示で変更した**（下記）。実データでの検証結果：

### 階層色の変更（指示書9-1「変更不可」からの変更・発注者承認済み）

| 階層 | 変更前（指示書9-1） | 変更後 | 変更後の彩度 |
|---|---|---|---|
| 第1 | 青 #0D50C8 / #E3EDFF | 青 **#062E7A / #B3D2FF** | 11% → 30% |
| 第2 | 赤 #C21807 / #FFE8E3 | 赤 **#9B0F06 / #FFB8AE** | 11% → 32% |
| 第3 | 緑 #0E7A2E / #DFF5E4 | 緑 **#06541D / #93E5A7** | 9% → 36% |
| 第4 | 紫 #7A1FA8 / #F2E4FB | 黄 **#5E4300 / #FFDF3D** | 9% → 76% |
| 第5 | 褐 #B05A00 / #FFEFD6 | 紫 **#4E0E73 / #DFAAF7** | 16% → 31% |

理由：**旧配色は背景の彩度が9〜16%しかなく**、文字色の色相は違っても
「どれも白に近い淡い色」で、一目では階層を見分けられなかった（発注者指摘）。
彩度を30〜76%に上げ、色相を離した。文字／背景のコントラストは全階層 5.2:1 以上（WCAG AA）、
文字／白は 8.5〜12.8:1（旧 4.9〜8.1:1）。実データの入れ子は最大4階層なので、
実際に目にするのは**青・赤・緑・黄**の4色。

色の実体は `docs/style.css` の `:root` の `--k1..--k5` / `--kb1..--kb5` の1か所だけ。
本文の着色も凡例も同じ変数を見るので、色を変えるときはここだけ直せばよい。

なお色分けモードで印刷すると背景色もそのまま出る（`print-color-adjust:exact`）。
インクを節約したい場合は「標準」モードで印刷する。

- **鉤括弧「」を括弧の深さの障壁として扱う必要があった。** 読替規定
  （`同条第八項中「純損失の金額（」とあるのは…`）は他法令の条文の**断片**を引用するため、
  括弧が開いたまま／閉じたままになる。素朴に深さを数えると**308か所**で深さが漏れ、
  以降の条文の色が総崩れになる。「」を障壁にすることで漏れは解消した（残る不整合140件は
  すべて「対応する開き括弧の無い `）`」で、地の文として出すため表示は壊れない）。
- **実データの括弧の入れ子は最大4階層**（「」を障壁とした実効値）。
  深さ0が55%、1が35%、2が8.8%、3が0.7%、4は9件のみ。
  **第5階層（紫）と6階層目からの循環は、現行の24法令には出現しない**（CSSは仕様どおり用意してある）。
  モックの措法99条（入れ子7階層）は架空条文だったため、指示書9-7に従い収録していない。

## 書体と文字の濃さ（2026-07-15 発注者指示で調整）

Windowsの游明朝・游ゴシックは太さ400だと極端に細く出るため、本文・UIとも
**Windows標準の BIZ UD書体**（ユニバーサルデザイン＝可読性のために設計された書体）を
優先する。iPad・Mac には BIZ UD が無いので Hiragino に落ちる。

- `--mincho`：BIZ UDPMincho Medium → Hiragino Mincho ProN → Noto Serif JP → Yu Mincho
- `--gothic`：BIZ UDPGothic → Hiragino Kaku Gothic ProN → Yu Gothic UI → Yu Gothic
- `--ink` #23262B → **#0E1116**、`--usuzumi` #6B7076 → **#383E45**
- 太さの下限を 500 に（`body` / `.honbun`）。指示書9-2の骨格600／括弧内400は、
  400が細すぎるため **700／500** に引き上げた（幹と枝の2段階の対比は保持）
- 目次の条見出しは 11.5px/400/#6B7076 → 12.5px/500/#0E1116（コントラスト比 5.0:1 → **18.5:1**）

実測：目次領域の「濃い画素の割合」は初版 0.4% → 6.8%（17倍）。

## ビューアの機能

- 初期メニュー（カード型ランチャー）⇄ ビューア（左：目次ツリー／右：条文）
- 法令別カラーを上部インデックス帯・カード・目次・条番号スタンプ・項番号バッジ・パンくずまで一貫
- 条文本文は明朝／UIはゴシック
- **条番号ジャンプ**：`法法22`／`所法56`／`措法66の6`。略称・正式名称・漢数字・全角数字・
  `第22条の2` 形式・三段の枝番（`措規22の11の3`）に対応
- **法令内検索**：条番号・見出しは即時。「本文も」ボタンで本文全文検索（チャンクを読み込む）
- **括弧色分け3モード**（色分け／標準／薄字）＋文字サイズ。localStorage に記憶
- 附則（改正法令ごと）・別表／様式のタブ
- URLハッシュで条を直接指せる（`#/houjin_act/a22`）＝ブックマーク・戻る操作に対応
- A4印刷CSS（ヘッダ・目次・ボタンを落とし、色分けは印刷にも反映）
- iPad Safari：768px/1024px でドロワー動作、入力時の自動ズーム抑止

## 出典・ライセンス

- 法令データ：**e-Gov法令検索（デジタル庁）** 法令API v2 より取得。
  [政府標準利用規約（第2.0版）](https://www.e-gov.go.jp/) に基づき出典を明示して利用。
- 通達（Phase 2）：**国税庁「法令解釈通達」** https://www.nta.go.jp/law/tsutatsu/
- 法令・通達は著作権法第13条により著作権の目的とならない。
- 本リポジトリのスクリプト・ビューアのコードは福島会計事務所に帰属。

法令データは加工して表示しています。正確な法令は必ず
[e-Gov法令検索](https://laws.e-gov.go.jp/) の原文および官報で確認してください。
現行法令のみを収録し、未施行の改正は扱っていません（指示書5-6）。
